JavaServerTM Faces (JSF) 1.0 EA3
Release Notes


For an updated, online version of these release notes, see Release Notes

Revised:02/24/2003

The purpose of this release is to introduce you to the JavaServer Faces framework and to provide you with an opportunity to give us feedback on the work we've done so far.

This Early Access release of JavaServer Faces provides an early alpha version of the JSF technology which is currently being developed under the JCP as JSR127. The expert group is actively working on the specification. This alpha release reflects a very early snapshot of a draft specification which has evolved significantly since the code freeze for this EA release.  This means that the APIs and designs contained in this release do not necessarily match what will be released in the upcoming releases of the specification.  This should be kept in mind when experimenting with this JSF technology.

What This Release Includes

     
  • html_basic.tld: A JSP custom tag library for building JSF applications that render to an HTML client.
  • jsf_core.tld: A JSP custom tag library for representing core actions independent of a particular RenderKit.
  • An API and reference implementation that provide tag handling, object management, rendering, validation, data conversion, and event processing.
  • A set of demos in the examples directory. Please note that these demos are provided as a proof-of-concept rather than a best-practices guide for writing JavaServer Faces applications.
  • JavaServer Faces, a tutorial to help you get started.

Known Issues With This Release

  • Be aware of bug . JSP with lots of custom tags crashes tomcat VM
  • Due to a bug in JSTL 1.0, you must have the jstl.jar and standard.jar in your WEB-INF/lib directory of each webapp.  It is not sufficient to place these jars in your tomcat/common/lib directory.
  • When saving tree in session, we don't support multiple concurrent  requests to faces app from one client.
  • SaveState in page option cannot be used if your web app has event handlers and validators because they are not serializable currently.
  • Component type is hard coded in component subclasses (ex: UICommand) as fully qualfied class names (ex: javax.faces.component.UICommand). However, a component's type is configurable in the renderkit's xml configuration file (HtmlBasicRenderKit.xml).
  • AttributeDescriptor support happens all the time, even though it's only used at design time.
  • Additionally, in the areas of functionality released in the EA, there are already significant changes in progress in the specification. Therefore, only experimental code should be developed against this release since it will not be compatibile with subsequent releases.
