JavaServerTM Faces (JSF) 1.0 Beta
Release Notes


For an updated, online version of these release notes, see Release Notes

Revised:20031205

The purpose of this release is to introduce you to the JavaServer Faces framework and to provide you with an opportunity to give us feedback on the work we've done so far.

This Beta release of JavaServer Faces technology provides a beta version of the JSF technology which is currently being developed under the Java Community Process(SM) ("JCP(SM)") as JSR127. The expert group is actively working on the specification, and you should expect to see continued evolution between this release and the final draft of the specification. Therefore, applications built on this release of JavaServer Faces technology are not guaranteed to work without modification on future releases.

What This Release Includes

     
  • html_basic.tld: A JSP custom tag library for building JavaServer Faces applications that render to an HTML client.
  • jsf_core.tld: A JSP custom tag library for representing core actions independent of a particular RenderKit.
  • An API and reference implementation that provide user interface components, model object management, pluggable rendering, server side validation, data conversion, event processing, page flow management, and custom extensibility for all of these things.
  • A set of demos in the samples directory. Please note that these demos are provided as a proof-of-concept rather than a best-practices guide for writing JavaServer Faces applications.
  • JavaServer Faces, a tutorial to help you get started.

Migrating From Beta to 1.0

This section can be used as a reference for bringing JavaServer Faces applications based on the Beta release into compliance with the 1.0 release of JavaServer Faces technology. This section may not cover all the changes to the technology that went into the Beta release. Please refer to the JavaServer Faces Specification for more details on the changes listed because the descriptions below only give a summary of the changes.

Migration of Configuration Files(faces-config.xml)

  • All "value-ref" elements should be replaced with with "value", "action-ref" with "action".
  • All expressions must be modified to follow the JSF EL expression syntax which means including "#{...}" delimiters.
  • <messageResources> element replaced by <resourceBundle>. Messages from XML files should be moved to resource bundles. Standard error messages can be replaced by custom error messages only for validation errors by adding the appropriate messageId and message to the resource bundles.

Migration of Java Classes

  • ValueChangedEvent renamed to ValueChangeEvent.
  • New state saving API allows developers to choose what state information to save if state saving method is "client"
  • The default value for "valid" on UIInput component is now true, so setValid() should be called only if validation fails.
  • SelectItemGroup is a new class that allows items in the list to be grouped.
  • Since UIForm now implements a NamingContainer, "id" should be fully qualified. So any calls to findComponent should be fixed. If a textfield with an id of "userName" is enclosed inside a form called "helloForm", parameter passed to findComponent would be "helloForm:userName" instead of just "userName" in EA4.
  • Tree and TreeFactory are now replaced with UIViewRoot component.
  • ActionEvent doesn't include a commandName anymore.
  • The javax.faces.application.Action API went away, along with the need to create a dummy getter to return an Action.

Other changes required for migration to Beta

  • MethodBinding expression(syntax: #{logonBean.userName}) enables referencing methods in a single backing bean class.
  • Components can be  bound to properties in the model via "binding" attribute and attributes can be manipulated directly.
  • New JSF EL expression syntax is #{logOnBean.userName}
  • JSF is not interoperable with JSTL iterator tag.
  • All JSP tag attributes are JSF EL enabled except "id", "var"
  • JavaServer Faces Reference Implementation now can output XHTML if the content type is set appropriately.
  • Because of a bug in UIData, literal values cannot be used with components nested inside a data_table tag. You have to bind the component value to a property in the model as a workaround.
  • To get logging output from JavaServer Faces RI, follow the instructions in README.
  • JSF now supports extension mapping/prefix mapping of a URL pattern to FacesServlet. For example *.jsf  for /faces/* can be mapped to FacesServlet.
  • Custom component classes should have their properties JSF EL-enabled where appropriate. Please refer to our Components demo tag handlers included with our samples for details on how to JSF EL enable component properties.(All custom components except TabbedPane are JSF EL enabled.)
  • Custom component classes should be modified to support saving and restoring state if you are going to save state in client.
  • Custom renderers should be modified to comply with changes to the ResponseWriter API.
  • Custom tags for component+renderer combinations need to change overrideProperties to setProperties, and should JSF EL-enable the attributes that correspond to JSF EL-enabled component properties+attributes.

Known Issues With This Release

  • the algorithm for scanning jars for faces-config.xml files will miss occurrences that reside outside of WEB-INF/lib, or on a custom ClassLoader. Workaround: make sure any faces-config files you want to be found reside inside of jars that reside in WEB-INF/lib.
  • UIData children that do not use VB expressions fail to maintain their values. In other words, if you have a literal value in a child of a UIData component, that value will be lost on postback. Workaround: always use a VB expression with the value attribute for UIData children.
  • We don't have a consistent, robust, and documentable logging story. We will for 1.0.

  • UIComponent.broadcast() fails to throw IllegalStateException if passed invalid Parameter.

      From the javadocs of UIComponent.broadcast():

   java.lang.IllegalStateException - if PhaseId.ANY_PHASE is
  passed for the phase identifier 

  No exception is currently thrown by the RI.
  • UIInput.validate() does not store zero-length String into previous property.

  • Incorrect type conversion for UISelectMany

    This class doesn't properly deal with conversion for primitives. Workaround: use wrapper classes instead of primitives for model objects.

  • <h:message> ignores locale set in <f:view>

        My system locale is French and if I do this...
    <f:view locale='en'>

        <h:output_text value='#{msgs.namePrompt}'/>
        <h:input_text id='name' value='#{user.name}'/>

        <h:message for='name'/>

    </f:view>

    <h:output_text> retrieves its output from my English resource bundle
    (as it should), but <h:message> stubbornly produces its error message
    in French (as it should not).

    • When using the non-JSP use-case, it's possible to use a VB expression for id attribute of UIComponent, but it should not be possible. Workaround: don't use VB expressions for the id attribute of UIComponent.

    The converterId attribute for <f:converter> is not VB enabled, and it should be.
